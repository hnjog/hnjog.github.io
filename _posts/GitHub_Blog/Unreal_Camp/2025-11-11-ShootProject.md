## Git 프로젝트 및 전략

[![Image](https://github.com/user-attachments/assets/e702ef6f-4888-45ef-9ca8-18cf6c55592f)](https://github.com/user-attachments/assets/e702ef6f-4888-45ef-9ca8-18cf6c55592f){: .image-popup}<br>

[7조 Z-WAVE](https://github.com/NbcampUnreal/5th_6th-Team7-CH3-Project?tab=readme-ov-file){:target="_blank"}<br>

사실 개인적으로 제일 익숙한 방식으로 진행하게 되었다<br>

- main의 경우는 원래는 프로토타입 완료 후 Merge하려 했으나<br>
  깜빡 잊고 진행하여, 단순히 마지막 Merge용 브랜치가 되었다<br>

- Dev의 경우, 메인 개발 브랜치로 활약했다<br>
  다만, Pull-Request 를 도입하지 못해 중간에 사고가 벌어진 적이 있었다<br>
  다음에 기회가 된다면 pr을 적용하여 Git 관련 사고를 예방해보고 싶다<br>
  (+ 코드 리뷰)<br>

- Feature를 적극적으로 만들고, 상황에 따라 원거리 브랜치에 Push하여<br>
  다양하게 사용하였다<br>
  (같이 버그 찾기 등)<br>
  - 개인적으로는 Local Branch로 사용하고, Rebase를 사용해<br>
    커밋을 깔끔히 남기는 것을 선호한다<br>

## KPT(Keep, Problem , Try) 회고

Keep : 잘하고 있는 점, 계속 했으면 좋겠다 싶음 점<br>
Problem : 문제가 있다 싶은 점, 변화가 필요한 점<br>
Try : 잘하고 있는 것을 더 잘하기 위해 / 문제를 해결하기 위해 시도할만한 점<br>

- 원래는 '도중'에 진행하는 방식<br>

그렇지만 현재로선 프로젝트가 '끝난' 상황이기에<br>
기준을 조금 다르게 잡아보았다<br>

Keep : 우리가 잘한 점, 다음 프로젝트에서도 하면 좋겠다 느낀 점<br>
Problem : 프로젝트에 이부분을 개선했다면 더 좋았다고 생각한 점 <br>
(문제 보다는 건설적인 피드백 과 의견에 가까움)<br>
Try : 이번 프로젝트를 바탕으로 다음 프로젝트 or 차후 스터디 등에서 시도할만한 점<br>

### Keep

- 자유롭게 의견을 말할 수 있는 분위기가 좋았다<br>
  : 하루에 회의를 2번 진행 (Planning / 일일 회고)<br>

- 유용한 시간 분배 및 팀 노션의 적극적인 활용<br>
  + Slack을 통한 소통<br>

- UML을 통해 구조를 잡고 시작한 것이 도움이 되었다<br>

- 체계화된 분업<br>

- 적극적인 팀 프로젝트 참여<br>

### Problem

- 비 개발 부분에 대한 작업 분배<br>
  ex) 기획 작업 분담<br>

- 커뮤니케이션<br>
  : '무엇'을 하자 에 대한 이야기는 많았으나<br>
    '어떻게 하자'에 대한 이야기는 적었던것 같다<br>


### Try

- 일정 인원에게 작업이 부담되었을 수 있음<br>
  따라서 차후 작업에는 '더 세밀한' 작업 분배 + 헬프 요청<br>
  - 발표 PPT 작성, 발표, 일정 관리는 프로그래머가 알아두면 확실히 좋은 부분<br>
  - 성장의 가능성이 될 수 있음<br>

- 자유롭게 의견을 말하되, 시간을 나누어 진행<br>
  - 아이디어 회의 시간 (자유로운 의견 토의)<br>
  - 구현에 대한 회의 시간 (구현에 대한 가능성 / 일정에 대한 고려)<br>

## 후기

팀장으로서 무언가를 적극적으로 해본 첫번째 프로젝트였다<br>
처음에는 룰렛으로 뽑힌 팀장이였으나, 기왕 하기로 한 이상 최선을 다하려 했다<br>

- 전반적으로 기획 + 영상 제작 만 다른 분께 맡기고<br>
  나머지는 직접 시도해보았다<br>
  - 프로젝트의 구조 제작 + UML<br>
  - 개발 일정 조율<br>
  - 분업과 협업 조율<br>
  - PPT 제작 및 발표<br>

- 전반적으로 리드 프로그래머에게 요구되는 자질이기도 하며<br>
  해당 사안들에 대하여 연습할 수 있는 기회라고 생각이 되었다<br>

- 실제로 해보면서, 왜 팀장님들이 바쁘고 일정을 늘 생각하는지를 알 수 있었다<br>
  - 개인 작업을 별도로 진행하면서<br>
    구조 + 개발 일정 조율이 솔직히 매우 어려웠다<br>
  - Switch를 쓰고 싶지 않았으나 결국은 개발 기간을 고려해 Enum을 사용하였다<br>

- 또, 바쁜 나머지 3주차의 월~화 부분은 일정 조율을 제대로 하지 못하였다<br>
  그렇기에 팀원들에게 수요일에서야 디버깅을 인식시켜 버그를 고치고 마무리했다<br>
  - 원래 목표는 수요일 이었으나, 실제 완성은 금요일<br>
  - 그래도 주말을 이용해 발표 자료를 만들 수는 있었다<br>
  - 발표 자료를 더 일찍 만들었으면, PPT도 더 잘 만들수 있었을텐데 하는 아쉬움이 있다<br>

- 기술적인 도전을 더 해봤어야 했나 싶다<br>
  - GAS를 넣지 않기로 하였는데, 강행을 했어야 했나? 라는 생각이 이제서야 든다<br>

다음에도 팀장을 할지 모르겠으나<br>
확실히 여러 부분을 고민하게 되었다<br>

- 단순히 공부용 프로젝트가 아니라<br>
  누군가에게 '발표'를 하기 위한 것<br>

- 그렇기에 '기술적'인 이해가 필요하며<br>
  그에 대하여 '서술'할 정도의 지식은 갖추어야 함<br>

- 아무리 잘 만들어도 잘 보여주지 못하면 인상을 남기기 힘들다<br>

- 아무리 잘 만들어도 TIL 작성을 안하면 남는것이 없을 수 있다<br>

- 다음에는 기술적인 도전을 좀 더 해보면서 남길만한 내용이 있는지 고려해야 겠다<br>
